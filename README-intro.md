# 🔍 API Test Automation & CI Pipeline
### QA Engineer (5+ years) → Automation & CI-driven Quality Engineering

## 👤 About This Project

본 프로젝트는 **5년 이상 수동 QA 경험을 가진 QA 엔지니어**가  
기존의 **Postman 기반 수동 API 테스트 프로세스**를

> **“반복 실행 가능한 자동화 + CI 기반 품질 게이트 + 운영 관점의 가시화”**

로 확장한 **API 테스트 자동화 프로젝트**입니다.

단순히 테스트를 자동으로 실행하는 수준이 아니라,  
**QA 관점에서 실제 현업에서 겪었던 문제들**을 해결하는 데 초점을 맞추었습니다.

---

## 🎯 Background (문제 인식)

5년간 QA 업무를 수행하며 다음과 같은 한계를 반복적으로 경험했습니다.

- Postman에서 **사람이 직접 호출 버튼을 누르는 테스트**
- “1회 통과”로는 검증되지 않는 **간헐적 API 실패**
- 테스트 결과가 **로그로만 남아 공유·추적이 어려움**
- 테스트가 실패해도 **즉시 인지되지 않는 구조**
- QA 테스트가 **CI / 배포 파이프라인과 분리**되어 운영되는 문제

이 프로젝트는 위 문제들을 **자동화 + CI 관점에서 구조적으로 해결**하는 것을 목표로 합니다.

---

## 🛠️ What I Did (QA 관점의 자동화 확장)

본 프로젝트에서 수행한 작업은 단순한 스크립트 작성이 아니라,  
**“QA 엔지니어가 품질을 어떻게 자동으로 보장할 수 있는가”**에 대한 설계입니다.

- Postman Collection을 **Newman 기반 자동 실행 구조**로 전환
- Iteration 증가를 통한 **API 안정성 테스트(반복 실행)**
- 실패 허용치(Threshold)를 도입한 **현실적인 품질 게이트**
- 로그 기반 실패 집계를 통한 **정량적 품질 판단**
- HTML 리포트를 통한 **테스트 결과 가시화**
- Slack 연동으로 **즉각적인 실패 인지**
- GitHub Actions CI 연동으로 **지속 가능한 테스트 파이프라인 구성**
- 정기 스케줄 실행을 통한 **운영 관점의 API 헬스 체크**

---

## 🧠 QA Engineer로서의 관점

이 프로젝트는 “개발자처럼 테스트 코드를 작성했다”가 아니라,

- **QA 경험에서 출발한 문제 정의**
- **품질 기준을 코드로 표현**
- **CI를 품질 게이트로 활용**
- **운영·커뮤니케이션까지 고려한 자동화**

를 목표로 설계되었습니다.

즉,  
> **수동 QA → 자동화 QA → CI 기반 Quality Engineering**  
으로 역할을 확장해 나가는 과정의 결과물입니다.

---

## 🧩 Why This Matters

이 프로젝트는 다음과 같은 질문에 대한 실무적 답변입니다.

- 반복 실행 시 간헐적으로 실패하는 API를 어떻게 검증할 것인가?
- 외부 요인이 있는 환경에서 CI를 어떻게 “쓸 수 있게” 만들 것인가?
- 테스트 결과를 누가, 언제, 어떻게 인지해야 하는가?
- QA 테스트를 배포 전/후 운영 프로세스에 어떻게 녹일 것인가?
